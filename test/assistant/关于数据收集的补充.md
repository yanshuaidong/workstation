# 数据收集层补充说明（已实现）

## 1. 当前实现状态

`database/fut_pulse` 现在已经不再只是“资金强度采集”。

当前真实行为是：

- 自动检查并创建 `fut_variety` / `fut_strength` / `fut_daily_close`
- `today` 在上传 `fut_strength` 后，自动通过 AkShare 同步当日 `fut_daily_close`
- `upload-only` 在推断为今日模式时，也会自动补当日 `fut_daily_close`
- 历史 close 通过独立命令 `close-history` 处理

对应的已落地文件：

```text
database/fut_pulse/
├── database/
│   ├── __init__.py
│   └── init_tables.py
├── close_price.py
├── pipeline.py
├── main.py
└── uploader/mysql.py
```

---

## 2. 这次补充的核心改动

### 2.1 统一建表入口

新增：

- `database/init_tables.py`

作用：

- 统一读取 `database/fut_pulse/.env`
- 统一建立 MySQL 连接
- 统一检查并创建三张基础表

基础表清单：

- `fut_variety`
- `fut_strength`
- `fut_daily_close`

### 2.2 新增 close 价格工具

新增：

- `close_price.py`

作用：

- 读取 `config/varieties.json`
- 使用内部固定映射表 `CLOSE_API_SYMBOL_MAP`
- 调用 AkShare 拉主连历史
- 将目标日期的 `close_price` 幂等写入 `fut_daily_close`

### 2.3 today 自动补 close

`pipeline.py` 现已接入：

1. 截图
2. OCR
3. 上传 `fut_strength`
4. 同步当日 `fut_daily_close`

这保证 assistant 在日常使用时，能直接拿到同一天的资金强度和收盘价。

### 2.4 历史 close 单独命令

`main.py` 现已新增：

```bash
python main.py close-history --days 30
python main.py close-history --start 2026-03-01 --end 2026-04-10
python main.py close-history --start 2026-03-01 --end 2026-04-10 --symbol rbm,cum
python main.py --dry-run close-history --days 5
```

这样历史 close 和截图 / OCR 链路完全解耦，补录和重试都更清晰。

## 3. 固定映射规则

这次没有采用运行时去读外部映射文件再动态匹配，而是直接把 `key -> api_symbol` 固定写在 `close_price.py` 里。

这样做的目的很明确：

- 避免大小写不一致造成运行时漏匹配
- 让 `fut_pulse` 的 close 同步逻辑完全自洽
- 如果未来增减品种，改动点只有两个：`varieties.json` 和 `close_price.py`

维护规则：

1. `config/varieties.json` 是主品种清单
2. `close_price.py` 内必须维护一份与之对应的固定映射
3. 如果 `varieties.json` 增加了新品种，但映射没补，启动时会直接报错终止

## 4. 当前命令行为

### `python main.py today`

行为：

- 采集主力 / 散户截图
- OCR
- 上传 `fut_strength`
- 自动同步当日 `fut_daily_close`

### `python main.py history ...`

行为：

- 采集历史截图
- OCR
- 上传 `fut_strength`

限制：

- 不自动补历史 `fut_daily_close`

### `python main.py upload-only`

行为：

- 使用现有 `data/result.json` 上传强度数据

补充规则：

- 如果推断为今日模式，会继续自动补当日 close
- 如果推断为历史模式，只补强度，不补历史 close

### `python main.py close-history ...`

行为：

- 不依赖截图 / OCR
- 直接通过 AkShare 回填 `fut_daily_close`

## 5. 当前表结构要点

### `fut_daily_close`

当前最关键的字段是：

- `variety_id`
- `trade_date`
- `close_price`
- `collected_at`

约束：

- 主键：`(variety_id, trade_date)`

写入策略：

- 不存在则插入
- 已存在则更新

因此重复跑同一天或重复回填同一段历史，不会产生重复脏数据。

---

## 6. 和 assistant 的关系

`test/assistant` 现在默认把以下两张表当作同一层基础行情：

- `fut_strength`
- `fut_daily_close`

因此这次补充不是“附加优化”，而是 assistant 正常运行的前置条件。

具体影响：

- `assistant_db.py` 取最新交易日时依赖 `fut_daily_close`
- `data_loader.py` 读取市场上下文时会 `JOIN fut_strength + fut_daily_close`
- 机械账户和 LLM 账户开平仓价格也依赖 `close_price`

---

## 7. 使用建议

推荐日常顺序：

1. `cd database/fut_pulse && python main.py today`
2. `cd test/assistant && python src/daily_run.py --signal-date ...`

如果是补历史研究数据，推荐顺序：

1. `python main.py history --start ... --end ...`
2. `python main.py close-history --start ... --end ...`
3. `python src/daily_run.py --start-date ... --end-date ...`

---

## 8. 一句话结论

当前 `fut_pulse` 已经正式把 `fut_daily_close` 提升为日常产出的一部分：表初始化统一收口、今日 close 自动补齐、历史 close 独立命令回填、全流程保持幂等和可重试。assistant 可以稳定地把 `fut_strength + fut_daily_close` 当作完整底座使用。
