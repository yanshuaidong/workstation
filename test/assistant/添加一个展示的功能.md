## 实施方案

### 核心难点

当前两套数据存在断层：
- K线数据：`hist_{symbol}` 动态表，通过 `contracts_main.symbol`（如 `cum`）索引
- 主力/散户数据：`fut_strength` 表，通过 `fut_variety.id` 索引
- 两者无现成外键关联，需要手动建立映射

---

### 解决方案：fut_variety 增加 contracts_symbol 字段

在 `fut_variety` 表新增一列 `contracts_symbol VARCHAR(20)`，存储对应的 `contracts_main.symbol`。
这是最直接、最可维护的映射方式，避免用名称字符串匹配的脆弱做法。

**数据库变更：**
```sql
ALTER TABLE fut_variety ADD COLUMN contracts_symbol VARCHAR(20) DEFAULT NULL;
-- 手动填写映射，例如：
UPDATE fut_variety SET contracts_symbol='cum' WHERE name='铜';
UPDATE fut_variety SET contracts_symbol='aum' WHERE name='黄金';
-- 依此类推，逐品种填写
```

### 后端改动（automysqlback/routes/assistant_routes.py）

新增两个接口：

**接口 A：品种列表**
```
GET /assistant/variety-list
功能：返回 fut_variety 中 contracts_symbol 不为空的品种
返回字段：id, name, contracts_symbol
```

**接口 B：品种K线 + 主力/散户指数**
```
GET /assistant/variety-kline
参数：
  variety_id   必填，fut_variety.id
  start_date   可选，默认近60天
  end_date     可选

逻辑：
  1. 查 fut_variety 获取 contracts_symbol
  2. 查 hist_{contracts_symbol} 获取K线（trade_date, open_price, high_price, low_price, close_price, volume）
  3. 查 fut_strength 获取同日期范围的 main_force, retail（WHERE variety_id = ?）
  4. 以K线日期为主，LEFT JOIN 指数数据，按 trade_date 升序返回

返回结构：
{
  "code": 0,
  "data": {
    "variety": { "id": 1, "name": "铜" },
    "kline": [
      { "trade_date": "2026-04-01", "open": 78000, "high": 79000, "low": 77500, "close": 78500, "volume": 12000 }
    ],
    "strength": [
      { "trade_date": "2026-04-01", "main_force": 0.65, "retail": -0.32 }
    ]
  }
}
```

### 注意事项

1. **映射需手动维护**：新增品种时需同步填写 `contracts_symbol`，否则该品种不会出现在选择器中
2. **日期对齐问题**：`fut_strength` 日期与K线日期可能不完全一致，合并以K线为主（LEFT JOIN），指数缺失日前端图表正常留空即可
3. **hist_ 表字段名确认**：字段推断为 `open_price`、`high_price`、`low_price`、`close_price`，写接口前需 `DESCRIBE hist_{symbol}` 确认实际字段名
