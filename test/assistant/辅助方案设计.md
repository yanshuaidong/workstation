# 整体想法

## 一、总体观点：

想法1（机械决策）和想法2（LLM 决策）**本期同时上线，双账户并行运行**。理由：

1. **归因问题**：如果只有 LLM 账户，赚了钱无法回答"是研究出来的规律有效？还是 LLM 的判断有效？"。两个账户同期跑才能做干净对比。
2. **工程复用度高**：数据层、信号层、模拟账户层、可视化层完全共用，想法2 只是在每日流程末尾额外生成一份提示词文件，增量极小。
3. **研究报告里的四条策略（策略1A/1B 高幅度主散共振做多/做空、策略2A/2B D7 主散共振做空/做多）已经具备机械落地的全部条件**，机械账户不依赖 LLM 可以独立运行，两者互不阻塞。

| 模块 | 职责 | 读/写 MySQL |
|------|------|------------|

| `strategies.py` | 封装 4 条策略：1A/1B 高幅度主散共振做多/做空、2A/2B D7 主散共振做空/做多；幅度分位使用**近 3 个月滚动窗口** | 写 `assistant_operations` |
| `paper_account.py` | 3 万本金、10 倍杠杆、按收盘价成交、无摩擦；管理持仓开平仓、浮盈浮亏；三层退出逻辑：① 趋势止盈（main_force 反向）② 硬止损（期货反向 5%/仓位亏损 50%）③ 最长持仓 5 天 | 读写 `assistant_positions` / `assistant_account_daily` |
| `decision_mechanical.py` | 想法1：规则引擎，读 operations 表，更新 paper account | 读 `assistant_operations`，写持仓/账户 |
| `decision_llm.py` | 想法2：生成包含完整上下文的提示词文件，供本地 AI 工具读取 | 读 MySQL，写 `logs/llm/YYYY-MM-DD-prompt.md` |
| `write_llm_decision.py` | CLI 工具，供本地 AI 调用，把一条 LLM 决策写入 MySQL | 写 `assistant_operations`（llm）+ `logs/llm/YYYY-MM-DD-calls.jsonl` |
| `daily_run.py` | 每日总入口：拉数据→算信号→生成操作→更新机械账户→生成 LLM 提示词 | 调用上述所有模块 |

### signals.py 使用的 7 个指标（来自 plan3_indicators.py）

**⚠️ 分位数统一使用近 3 个月滚动窗口（约 63 个交易日）**：所有用到 p10/p20/p50/p80/p90/p25/p75 的地方，均只取当日往前 63 个交易日的数据计算，不用全历史。原因：若 3 年前出现过极端值（如 200），而近期 main_force 一直在 -30～30 之间，用全历史分位会让当前正常区间永远落在"中性"，策略信号永远无法命中极端区域，彻底失效。滚动窗口保证分位阈值始终反映**近期市场的相对强弱**。（初期会没有那么多数据，但程序不要报错，不满3个月的以获取到的不满3个月的数据来计算）

| # | 指标名 | 触发条件（LONG / SHORT） | indicator_value 含义 |
|---|--------|--------------------------|----------------------|
| 1 | **MF_Edge3** | LONG：`main_force` 严格 3 连升；SHORT：严格 3 连降 | 起始位置区域：`极端底部 / 普通底部 / 中性偏高`（LONG）或 `极端顶部 / 普通顶部 / 中性偏低`（SHORT）；区域由**近 3 个月滚动 p10/p50/p90** 决定 |
| 2 | **MF_Accel** | 在 3 连同向基础上，判断第2步变化幅度是否大于第1步 | `加速(+x.xx)` 或 `减速(-x.xx)`，数值为 `\|delta2\| - \|delta1\|`；无分位，不受此影响 |
| 3 | **MF_Duration** | 连续同向恰好达到 3 / 5 / 7 天时（同一段趋势只在刚好满足时触发一次） | `D3` / `D5` / `D7`；纯计数，无分位 |
| 4 | **MS_Divergence** | LONG：主力 3 连升 + 散户同期均值**下降**（主散背离）；SHORT：主力 3 连降 + 散户同期均值**上升** | `强共振(x.xx)` / `弱共振(x.xx)`，数值为背离强度分数；判断方向性，无分位 |
| 5 | **MF_Magnitude** | 严格 3 连同向，按总变化幅度（`\|v2-v0\|`）分三档 | `高幅度(x.xx)` / `中幅度(x.xx)` / `低幅度(x.xx)`；阈值为**近 3 个月滚动 p25/p75** |
| 6 | **MF_BreakZone** | 3 连同向中，判断是否从极端区域边界穿越出来 | `穿越底部边界` / `极端区内震荡` / `普通区间3连升`（LONG 侧）；边界由**近 3 个月滚动 p10/p90** 决定 |
| 7 | **Composite_Score** | 连续同向 ≥ 3 天即触发，综合 5 项得分加权：持续天数（30%）+ 加速度（20%）+ 边缘区启动（20%）+ 主散背离（20%）+ 幅度强度（10%） | 综合得分数值，约 0.0～2.0；其中边缘区启动和幅度强度子项的分位阈值同样用**近 3 个月滚动**计算 |

> **注**：上述所有指标均同时生成 LONG 和 SHORT 两个方向的信号。`signals.py` 在每日运行时，对当日数据触发上述判断，并将结果写入 `assistant_signals` 表。策略层（`strategies.py`）只使用其中 MF_Magnitude 和 MS_Divergence + MF_Duration 的组合来触发 4 条交易策略，其余指标用于信号面板展示和后续研究。

## 四、MySQL 新增表设计

```sql
-- 每日信号记录
CREATE TABLE assistant_signals (
    id           INT AUTO_INCREMENT PRIMARY KEY,
    signal_date  DATE NOT NULL,
    variety_id   INT NOT NULL,
    variety_name VARCHAR(20),
    indicator    VARCHAR(40) NOT NULL,   -- MF_Magnitude / MS_Divergence 等
    direction    VARCHAR(10),            -- LONG / SHORT
    triggered    TINYINT(1) DEFAULT 0,
    strength     FLOAT,
    extra_json   JSON,
    created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    KEY idx_date_variety (signal_date, variety_id)
);

-- 策略操作建议
CREATE TABLE assistant_operations (
    id             INT AUTO_INCREMENT PRIMARY KEY,
    signal_date    DATE NOT NULL,
    variety_id     INT NOT NULL,
    variety_name   VARCHAR(20),
    strategy       VARCHAR(40) NOT NULL,   -- 1A / 1B / 2A / 2B
    direction      VARCHAR(10) NOT NULL,   -- LONG / SHORT
    extra_json     JSON,                   -- 触发时各指标数值
    account_type   VARCHAR(20) DEFAULT 'mechanical',  -- mechanical / llm
    executed       TINYINT(1) DEFAULT 0,
    created_at     DATETIME DEFAULT CURRENT_TIMESTAMP,
    KEY idx_date (signal_date)
);

-- 模拟持仓
CREATE TABLE assistant_positions (
    id             INT AUTO_INCREMENT PRIMARY KEY,
    account_type   VARCHAR(20) NOT NULL,
    variety_id     INT NOT NULL,
    variety_name   VARCHAR(20),
    direction      VARCHAR(10) NOT NULL,
    open_date      DATE NOT NULL,
    open_price     FLOAT NOT NULL,
    close_date     DATE,
    close_price    FLOAT,
    size_pct       FLOAT DEFAULT 0.3,     -- 仓位比例
    status         VARCHAR(10) DEFAULT 'open',  -- open / closed
    pnl_pct        FLOAT,                 -- 平仓后填入
    strategy       VARCHAR(40),
    created_at     DATETIME DEFAULT CURRENT_TIMESTAMP,
    KEY idx_account_status (account_type, status)
);

-- 账户每日净值
CREATE TABLE assistant_account_daily (
    id           INT AUTO_INCREMENT PRIMARY KEY,
    record_date  DATE NOT NULL,
    account_type VARCHAR(20) NOT NULL,
    equity       FLOAT NOT NULL,          -- 当日账户净值
    cash         FLOAT NOT NULL,
    position_val FLOAT DEFAULT 0,
    daily_pnl    FLOAT DEFAULT 0,
    created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_date_account (record_date, account_type)
);
```

## 七、想法2（LLM 决策）的实现方式

**不走远程 API**，使用本地 AI 工具（Claude Code / Codex 等），流程如下：

```
daily_run.py
    ↓ 生成
logs/llm/YYYY-MM-DD-prompt.md   ← 包含全部上下文的提示词文件
    ↓ 用户手动触发
本地 AI 工具（Claude Code 等）读取提示词，做出决策
    ↓ AI 调用
src/write_llm_decision.py [CLI 工具]   ← AI 用这个工具把决策写入 MySQL
    ↓ 写入
assistant_operations（account_type='llm'）
assistant_positions（account_type='llm'）
```

**`decision_llm.py` 的职责**（只负责生成提示词文件）：
- 从 MySQL 读取当日触发的操作建议（机械策略生成的 operations）
- 为每个候选品种拼出近 10 日 main_force / retail / close 数据表
- 读取当前 LLM 账户持仓状态
- 将研究结论 + 数据 + 账户状态组装成 Markdown 提示词文件，写到 `logs/llm/YYYY-MM-DD-prompt.md`
- **不做任何决策**，只生成上下文

**`write_llm_decision.py` CLI 工具**（供本地 AI 调用，覆盖全部数据库操作）：

AI 读取提示词后，对每个候选品种调用此工具，工具负责一次性更新所有相关表，AI 无需直接操作数据库。

```bash
# 开仓
python src/write_llm_decision.py \
  --action open_long \
  --variety_id 28 \
  --variety_name rb \
  --size_pct 0.3 \
  --strategy "1A_high_mag_ms_resonance_long" \
  --reasoning "主力3连升且幅度p85，散户反向，触发策略1A..."

# 平仓
python src/write_llm_decision.py \
  --action close \
  --variety_id 28 \
  --variety_name rb \
  --reasoning "主力今日下降，趋势结束，触发趋势止盈..."

# 持仓不动（仅记录）
python src/write_llm_decision.py \
  --action hold \
  --variety_id 28 \
  --variety_name rb \
  --reasoning "信号尚未反向，继续持有..."
```

工具根据 `--action` 执行不同操作，**一次调用更新所有相关表**：

| action | assistant_positions | assistant_account_daily | assistant_operations | 归档日志 |
|--------|--------------------|-----------------------|---------------------|---------|
| `open_long` / `open_short` | 写入新持仓（status=open） | 更新当日净值 | 写入操作记录 | ✓ |
| `close` | 更新持仓（status=closed，填写 close_price / pnl_pct） | 更新当日净值 | 写入操作记录 | ✓ |
| `hold` | 不操作 | 不操作 | 不操作 | ✓ |

**生成的提示词文件结构**：
```
## 研究结论（固定背景知识）
[四条核心策略的触发逻辑、胜率数据]

## 今日触发信号（YYYY-MM-DD）
[品种 / 策略 / 方向 / 触发指标值 表格]

## 候选品种近10日数据
[每个触发品种的 main_force / retail / close 数值表]

## 当前 LLM 账户状态
[持仓列表 / 可用资金 / 近期盈亏]

## 操作指令
请对每个候选品种决定：open_long / open_short / close / hold，
然后调用 write_llm_decision.py 工具写入决策。
每条决策必须填写 reasoning 字段。
```

## 八、容易踩坑的点

1. **信号稀缺是正常的**：D7 SHORT 全部 28 天只有 47 个信号，平均每天不到 2 个。不要为了"每天有操作"而放宽触发条件，那就把研究成果白做了。
2. **样本外验证才是真正的考验**：报告的胜率基于 2026 年 3~4 月偏多头市场，实战即样本外。**实验阶段暂不设置策略暂停告警**，积累一段时间样本外数据后，再评估是否需要引入自动熔断机制。
3. **多策略冲突**：同一品种策略说做多、另一条说做空时，MVP 阶段最简单的规则：同品种只允许单方向持仓，新信号与已有持仓方向冲突则忽略。
4. **人工确认开关**：即使是机械决策，强烈建议前三个月保留"人工 confirm 才真正记录到模拟账户"的步骤。全自动可以等到样本外数据跑满 3 个月、策略稳定再放开。
5. **持仓退出规则（三层，按优先级依序检查）**：
   - **① 趋势止盈（主力离场）**：做多持仓时，`main_force` 当日较前一日**下降**则触发退出；做空持仓时，`main_force` 当日较前一日**上升**则触发退出。逻辑：跟随主力，主力不再延续方向就离场，保住已有收益。
   - **② 硬止损（期货反向 5%）**：10 倍杠杆下，期货价格反向运动 5% = 仓位亏损 50%，无论趋势信号如何均**强制平仓**，防止大幅亏损。
   - **③ 最长持仓 5 天**：前两条均未触发时，持满 5 个交易日自动平仓（对应研究报告的回测持有期上限）。
6. **不要把机械账户和 LLM 账户混在一起**：两个独立账户、独立净值，写入不同的 account_type，这样才能做干净对比。
7. **多信号并发时仓位优先级**：同日触发多个信号，按 Composite_Score 从高到低排序，强者优先填满空位（每笔 30% 仓位，3 万本金最多 3 个仓位同时持仓）。已有持仓的品种不重复开仓，等待条件①②③触发退出后再考虑新信号。不频繁切换，给主力资金足够时间运作。
8. **滚动分位数窗口**：**信号层和策略层所有涉及分位数的计算，一律使用近 3 个月（约 63 个交易日）滚动窗口，不能用全历史**。受影响的有：MF_Edge3 的区域判断（p10/p50/p90）、MF_Magnitude 的幅度分档（p25/p75）、MF_BreakZone 的边界判断（p10/p90）、Composite_Score 内的边缘区和幅度子项、以及 strategies.py 中策略1A/1B 的幅度分位。原因：若历史上出现过极端值，全历史分位会将当前正常区间永远归入"中性"，导致极端区域信号永远无法命中，策略完全失效。
